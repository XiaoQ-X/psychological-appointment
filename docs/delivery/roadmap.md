# 后续增强路线图

生成时间：2026-06-01

## 当前原则

当前交付基线已经完成三端 UI 原型复原、API 接入和关键联动回归。后续增强必须作为独立阶段推进，不混入当前验收结论。

增强阶段仍需遵守：

- 不随意重做已验收页面 UI。
- 学生端和咨询师端继续保持移动端小程序风格。
- 管理员端继续保持 PC 后台风格。
- API 增强需要先明确业务规则，再设计最小接口。
- 数据库改动必须单独评审并生成迁移，不夹带在 UI 修复中。

## P1 增强

| 项目 | 当前状态 | 建议目标 | 验收标准 |
|---|---|---|---|
| 学生端推荐问卷结果保存 | 当前保留原型问卷和推荐展示，未设计独立保存接口。 | 保存学生推荐问卷答案、推荐结果和推荐时间，供后续回看或优化推荐算法。 | 学生提交问卷后结果可持久化；刷新后仍可查看最近推荐结果；不改变现有推荐页布局。 |
| 学生端自助转介申请 | 当前转介闭环以咨询师发起为主，学生端自助转介仍为原型占位流程。 | 明确学生主动申请转介的审批规则、消息通知和管理员/咨询师处理方式。 | 学生可提交转介申请；咨询师或管理员可处理；状态和消息三端同步；不破坏现有咨询师发起转介闭环。 |

## P2 增强

| 项目 | 当前状态 | 建议目标 | 验收标准 |
|---|---|---|---|
| 咨询师文章独立编辑更新接口 | 咨询师文章编辑页已可展示，新建流程可用；历史文章独立更新接口仍可增强。 | 增加咨询师维护历史文章的更新能力。 | 支持编辑历史文章并保存；学生端文章列表和详情同步更新；移动端编辑页结构不变。 |
| 全页面截图级视觉回归 | 当前已完成原型映射、构建、API 和关键页面浏览器抽查。 | 建立三端全页面截图基线，逐页留档，降低后续改动造成视觉回归的风险。 | 每个页面都有截图记录；低于 4 分的页面不得通过；报告列明差异和修复建议。 |
| 上线部署与安全加固 | 当前为本地开发和验收交付。 | 准备生产部署、环境变量、安全策略、日志策略和备份策略。 | 生产启动文档完整；敏感配置不入库；数据库备份和恢复流程可执行。 |

## 建议执行顺序

1. 人工验收当前交付基线。
2. 固化当前可交付版本。
3. 进入 P1：学生端推荐问卷结果保存。
4. 进入 P1：学生端自助转介申请。
5. 进入 P2：咨询师文章独立编辑更新接口。
6. 进入 P2：全页面截图级视觉回归。
7. 进入 P2：上线部署与安全加固。

## 每批增强的交付要求

每批增强都需要单独产出：

- 需求说明。
- API 或数据结构变更说明。
- 修改文件清单。
- 构建和测试结果。
- 三端影响范围。
- 验收报告。

如果某批涉及数据库 schema，需要额外提供：

- migration 文件。
- seed 是否需要更新。
- 旧数据兼容说明。
- 回滚说明。
